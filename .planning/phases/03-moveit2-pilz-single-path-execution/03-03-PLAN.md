Current _goal_callback structure (to be extended):
  1. _is_active check → REJECT
  2. _is_executing lock acquire → REJECT if already executing
  3. TrajectoryGoalDTO.from_ros_msg(goal) in try/except → REJECT on ValidationError/ValueError
  4. return GoalResponse.ACCEPT
  Phase 3 inserts NEW step 3 (before from_ros_msg):
    for each path in goal.paths where path.motion_type == 'CIRC':
      if path.circ_type not in ('interim', 'center'):
        log ERROR with path.path_id and path.circ_type value
        release _is_executing flag
        return GoalResponse.REJECT
  The existing DTO validation becomes step 4.

Current _execute_callback structure (to be replaced):
  - Calls from_ros_msg, groups paths, sends stub feedback per GROUP
  Phase 3 replaces the inner loop with path-level flatten + plan + execute:
    for group in groups:
      for path in group:
        publish feedback(executing, [path.path_id])
        plan_result = self._planner_service.plan(path)
        if not plan_result.success:
          log ERROR; abort; return Result(success=False, ...)
        exec_status = self._moveit.execute(plan_result.trajectory, controllers=[])
        if not exec_status:
          log ERROR with path.path_id; abort; return Result(success=False, ...)
        publish feedback(completed, [path.path_id])
        completed_ids.append(path.path_id)
    succeed; return Result(success=True, trajectory_paths_completed=completed_ids)

  CRITICAL: execute() signature is moveit.execute(trajectory, controllers=[])
  — NO blocking kwarg (Research §Pitfall 4).
  NOTE: D-09 specified 'blocking=True' in the execute() call. Research Pitfall 4
  confirms this kwarg does not exist in the MoveItPy.execute() C++ binding and would
  raise TypeError at runtime. This plan removes blocking=True, correcting D-09.

  The outer except/finally pattern is unchanged from Phase 2:
    except Exception: log ERROR; abort; return failure result
    finally: release _is_executing flag
