<decisions>
## Implementation Decisions

### TrajectoryPath.msg — Field Design
- **D-01:** CIRC-specific fields are **flat** in `TrajectoryPath.msg` (not extracted to a sub-message). Fields `circ_type` (string) and `circ_point` (`geometry_msgs/Point`) are always present; ignored for LIN/PTP.
- **D-02:** `circ_type` is a **string field with string constants** defined in the `.msg` file: `string CIRC_TYPE_INTERIM="interim"` / `string CIRC_TYPE_CENTER="center"`. Wire format remains string; named constants available in generated Python code.
- **D-03:** `circ_point` is `geometry_msgs/Point` (no frame). Its frame is assumed to match the `target_pose` header frame.
- **D-04:** Each path has a single `geometry_msgs/PoseStamped target_pose` (one target pose per path, as required). No additional via-pose field.
- **D-05:** Each `TrajectoryPath.msg` includes a `string tool_frame` field (default empty → treated as `tool0` at planning time). This allows per-path TCP specification — e.g., using the TF frame of an attached object as the IK target for cartesian motion. When empty, node defaults to `tool0`.

### ManageScene — Multiple Service Files
- **D-06:** Scene management is implemented as **5 separate `.srv` files** (not a single multiplex service): `AddObject.srv`, `AttachObject.srv`, `DetachObject.srv`, `RemoveObject.srv`, `ModifyAcm.srv`.
- **D-07:** All scene service responses include: `bool success`, `string error_message`, `string object_id`. The `object_id` echoes the object acted on (or the newly generated ID for `AddObject`).

### AddObject.srv
- **D-08:** Object identity is **server-generated UUID4** — callers do NOT provide an object_id in the request. The UUID4 `object_id` is generated by the server and returned in the response. Callers must store this ID for subsequent attach/detach/remove/modify_acm calls.
- **D-09:** Geometry is expressed using the **standard `shape_msgs/SolidPrimitive` field** for primitives (BOX, SPHERE, CYLINDER with its `dimensions[]` array) plus a separate `string mesh_file_path` field for meshes. When `mesh_file_path` is non-empty, the SolidPrimitive field is ignored (mesh mode). When empty, SolidPrimitive is used (primitive mode).
- **D-10:** Object pose is `geometry_msgs/PoseStamped` (includes `frame_id` in the header so callers can express the pose in any TF frame; MoveIt2 transforms to the planning frame).

### AttachObject.srv
- **D-11:** Request fields: `string object_id` (UUID4 returned from AddObject), `string link_name` (e.g., `tool0`), `geometry_msgs/Pose attach_pose` (pose of the object relative to the link frame at attach time — explicit, not inferred from current scene state).
- **D-12:** Attaching an object does NOT automatically modify the ACM. Callers must explicitly call `ModifyAcm.srv` to allow/disallow collisions between the attached object and adjacent links if needed.

### ModifyAcm.srv
- **D-13:** Request uses **pair-list encoding**: `string[] object_ids_a`, `string[] object_ids_b`, `bool allowed`. Sets the ACM entry for every pair (a_i, b_j) across the two lists.

### Python Module Layout
- **D-14:** Sub-packages (`models/`, `enums/`, `utils/`, `services/`) are **truly empty stubs** — only `__init__.py` files with BSD-3-Clause license headers. No skeleton class files, no `__all__` declarations. Downstream phases create all files from scratch.
- **D-15:** `ur_movement_controller.py` is **NOT** created in Phase 1. It is a Phase 2 deliverable.

### Phase 1 Smoke Test
- **D-16:** The smoke test verifies:
  1. All generated ROS2 interface types are importable from Python (`from movement_controller.action import ExecuteTrajectory`, all srv types, all msg types).
  2. All Python sub-packages are importable (`from movement_controller.models import *`, etc.).
  - Does NOT check BSD header presence or file structure; that is a human review concern.

</decisions>

<specifics>
## Specific Ideas

- The `tool_frame` per-path field emerged from a concrete use case: attaching a `tool0_extension` object to `tool0` and then performing cartesian motion to the tip of that extension. MoveIt2 publishes a TF frame for attached objects; setting `tool_frame` to the attached object's UUID4 ID routes PILZ's IK to that frame.
- CIRC path with `circ_type = "center"` means `circ_point` is the center of the arc circle. With `circ_type = "interim"` it is a via-point on the arc. Both must be validated in Phase 3 tests.
- `ModifyAcm.srv` pair-list encoding: `object_ids_a = ["tool_extension_uuid"]`, `object_ids_b = ["tool0"]`, `allowed = true` is the standard call after AttachObject for a tool extension.

</specifics>
