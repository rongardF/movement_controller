## Objective

Add per-path speed and acceleration cap enforcement to `URMovementController._goal_callback`.
When `constraints.max_cartesian_speed > 0.0` and any path in the goal exceeds it, the goal is
rejected immediately with `GoalResponse.REJECT` and a D-07-format error message. Same for
`max_acceleration`. This runs in the `_goal_callback` (before `_execute_callback`), so no
planning resources are consumed on invalid goals.

Purpose: Operator-configured speed caps are enforced at the action boundary (CON-05, D-07).

Output: Updated `_goal_callback` in `ur_movement_controller.py` with speed cap loop.

Note: This plan runs parallel to 05-02. Both are Wave 2 but touch different files:
05-02 modifies only `pilz_planner_service.py`; this plan modifies only `ur_movement_controller.py`.

  <action>
    In `_goal_callback`, after the `except (ValidationError, ValueError)` block that rejects on
    DTO failure (and before the final `return GoalResponse.ACCEPT`), insert a speed cap
    enforcement block.

    The block should:
    1. Guard with: `if self._constraint_config is not None:`
    2. Inside the guard, iterate `for path in self._trajectory_goal.paths:` (these are
       `TrajectoryPathDTO` objects, already validated by `from_ros_msg`).
    3. Cartesian speed check: if both `path.cartesian_speed > 0.0` and
       `self._constraint_config.max_cartesian_speed > 0.0` and
       `path.cartesian_speed > self._constraint_config.max_cartesian_speed`:
       - Build error message using exact D-07 format:
         `f"Path '{path.path_id}' cartesian_speed {path.cartesian_speed} m/s exceeds node maximum {self._constraint_config.max_cartesian_speed} m/s (constraints.max_cartesian_speed)"`
       - `self.get_logger().error(msg)`
       - `self._trajectory_goal = None`
       - Reset `self._is_executing = False` inside `with self._executing_lock:`
       - `return GoalResponse.REJECT`
    4. Acceleration check: same structure, replacing `cartesian_speed` with `acceleration`
       and `constraints.max_cartesian_speed` with `constraints.max_acceleration` in the message:
       `f"Path '{path.path_id}' acceleration {path.acceleration} m/s² exceeds node maximum {self._constraint_config.max_acceleration} m/s² (constraints.max_acceleration)"`

    Note: The loop is inside the `if self._constraint_config is not None:` guard. If
    `_constraint_config` is None (node not yet configured with constraints), the check is
    skipped and ACCEPT is returned as before.

    The final `return GoalResponse.ACCEPT` at the end of `_goal_callback` remains unchanged.
  </action>

  <acceptance_criteria>
    - Source: `_goal_callback` contains `if self._constraint_config is not None:` guard before speed loop
    - Source: cartesian_speed error message contains literal substring `'cartesian_speed'` and `'(constraints.max_cartesian_speed)'` matching D-07 format
    - Source: acceleration error message contains literal substring `'acceleration'` and `'(constraints.max_acceleration)'`
    - Source: both rejection branches set `self._trajectory_goal = None` and reset `self._is_executing = False` inside `with self._executing_lock:` before returning REJECT
    - Source: the final `return GoalResponse.ACCEPT` is still present after the new block
    - Behaviour: node with `max_cartesian_speed=0.5` and goal with `path.cartesian_speed=0.8` → `_goal_callback` returns `GoalResponse.REJECT`
    - Behaviour: node with `max_cartesian_speed=0.0` (default) and goal with `path.cartesian_speed=0.8` → `_goal_callback` returns `GoalResponse.ACCEPT` (0.0 = unconstrained)
    - Behaviour: node with `max_cartesian_speed=1.0` and goal with `path.cartesian_speed=0.5` → `_goal_callback` returns `GoalResponse.ACCEPT` (within limit)
    - Behaviour: `_constraint_config is None` → no speed check, returns `GoalResponse.ACCEPT` as before
  </acceptance_criteria>

## Security

No external I/O. Speed values come from the already-validated `TrajectoryGoalDTO` (Pydantic v2)
and from `ConstraintConfigDTO` (Pydantic v2). No arithmetic that could overflow or cause injection.

## Success Criteria

- Goal with `path.cartesian_speed > max_cartesian_speed` is rejected with `GoalResponse.REJECT`
- Goal with `path.acceleration > max_acceleration` is rejected with `GoalResponse.REJECT`
- Goal with `max_cartesian_speed=0.0` (default) is never rejected for speed regardless of path value
- Rejection resets `_is_executing = False` so the next goal is not permanently blocked
- `return GoalResponse.ACCEPT` still exists at the end of `_goal_callback`
